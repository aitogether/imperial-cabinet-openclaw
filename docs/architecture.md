# 架构详解 · 内阁都察制
「预计适合的使用者是谁」（一人公司 / 小团队 / 熟悉 OpenClaw 的开发者）

## 核心理念

内阁都察制采用中国传统官制隐喻来组织多 Agent 协作。核心原则是：

> **人类只做决策，AI 负责执行。一个入口进来，一个声音出去。**

## 组织架构

```
                    ┌─────────────┐
                    │   皇帝/用户  │
                    │  (唯一决策者) │
                    └──────┬──────┘
                           │ 只对御前首辅下旨
                    ┌──────▼──────┐
                    │  御前首辅    │
                    │ (调度中枢)   │
                    └──────┬──────┘
                           │ 按任务类型路由
           ┌───────────────┼───────────────┐
           │               │               │
    ┌──────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
    │    内阁     │ │   御史台    │ │    东厂     │
    │  (拆解规划) │ │ (审核封驳)  │ │ (监控审计)  │
    └──────┬──────┘ └──────┬──────┘ └─────────────┘
           │               │
           │ 审核通过后     │
    ┌──────▼──────┐        │
    │   八部执行   │ ◄──────┘
    └─────────────┘
```

## 角色定义

### 皇帝（用户）

唯一的真人决策者。只做「批示」与「拍板」，不亲自写命令和代码。所有指令通过御前首辅传达。

### 御前首辅（Grand Secretary）

- **技术代号**: `main` / `grand_secretary`
- **职责**: 用户唯一接口，多 Agent 调度中枢
- **不做**: 不长期亲自承担具体业务
- **要做**: 统筹、拆解、规划、路由、监督、汇报

### 八部

| 部门 | 代号 | 职责一句话 |
|------|------|-----------|
| 内阁 | cabinet | 总体协调与路线规划 |
| 工部 | engineering | 技术选型与系统架构 |
| 户部 | finance | 算力与 API 成本统计 |
| 兵部 | security | 安全策略与权限设计 |
| 吏部 | ops | 流程与协作规范 |
| 礼部 | content | 对外文案与知识整理 |
| 刑部 | compliance | 隐私 / 版权 / 政策红线检查 |
| 翰林院 | docs | 技术文档与教程 |

### 一厂：东厂（monitor）

监控与审计。记录关键操作、采集指标、异常检测、维护记忆总账、定期复盘。

### 御史台（censor）

审核与封驳。审核各部关键产出质量，可封驳打回重做，维护整体质量与风险底线。

## 任务流转示意

一个典型任务的完整流转路径：

```
1. 皇帝下旨
   "我想做一个自动化内容发布流水线"
        │
2. 御前首辅接收，澄清约束与目标
   "您的目标是日更？每周？需要哪些平台？"
        │
3. 内阁拆解任务
   拆成：选题模块、资料收集、写稿、审核、发布、复盘
   输出：里程碑计划 + 验收标准
        │
4. 御史台审核（高风险任务）
   审核方案完整性、风险、资源合理性、可回滚性
   → 通过：批准执行
   → 封驳：打回内阁重新规划
        │
5. 各部执行
   - 工部：写自动化脚本
   - 礼部：写发布模板
   - 兵部：配置 API 密钥安全存储
   - 吏部：设计 SOP 流程
        │
6. 东厂记录
   操作日志、耗时统计、异常检测
        │
7. 御前首辅汇总，回报皇帝
   "结论：流水线已搭建完成。
    证据：脚本路径 / 测试日志 / 首次发布记录。
    下一步：是否需要接入更多平台？"
```

## 关键设计决策

### 为什么用「约定」而非「强制」？

本框架采用 **渐进式严格** 策略：

- **初期**: 用 Markdown 文件定义规则和约定，依赖御前首辅自觉执行
- **中期**: 东厂定期审计，检查约定是否被遵守
- **后期**: 逐步引入程序化状态机和看板

这样做的好处是：
1. 零额外依赖，不需要部署后端
2. 可以快速启动，不需要先写一堆代码
3. 根据实际需要逐步加强约束

### 为什么御史台是独立的？

审核和执行分开，避免「自己审自己」的盲区。御史台只做审核，不做执行；八部只做执行，不做审核。

### 为什么皇帝只和御前首辅对话？

单一入口原则：
- 减少人类的认知负担（只需要记住一个入口）
- 确保所有任务都被统一调度（不会出现"左脚踩右脚"的情况）
- 御前首辅可以做全局视角的优先级判断
