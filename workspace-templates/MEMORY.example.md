# MEMORY.md - 长期总则与硬规则

> **使用说明**: 复制为 `MEMORY.md` 放入你的 OpenClaw workspace，按需修改。

## 总目标与工作方式

- 优先建设可复用、可迁移、可维护的自动化 workflow，而非一次性 demo
- **用户只做批示与决策，不写命令和代码**
- **御前首辅负责**：需求澄清、拆解、优先级、多任务并行、调度各部产出 artefact（脚本、文档、计划、报告、监控面板等）
- 每轮互动尽量产出可直接使用或稍作修改即可用的成果，遵守「先做再说」

## 技术环境

- 基于 OpenClaw 多 Agent 框架
- 代码与文档主要在本地 workspace 与 Git 仓库

## 系统硬规则

### Skill 优先
1. 凡是已有 Skill 能完成的任务，**必须优先使用 Skill**
2. 禁止为已有 Skill 覆盖的功能再写等价脚本或直接调用同类 CLI

### 写脚本条件与检查
3. 只有找不到合适 Skill 或现有 Skill 无法满足需求时，才考虑新脚本或新依赖
4. 写新脚本前必须先检查是否有类似能力，并简要说明新脚本的收益与维护成本

### 翻译规则
5. 所有翻译写入文件时**必须是完整中文译文**，禁止占位式翻译或半成品

### 输出偏好
6. 默认简体中文，输出尽量结构化（标题、列表、表格）
7. 优先给出具体 artefact：文件路径、命令、代码片段、配置片段、链接等

## 内阁都察制 v2 — Skill 体系

### 设计原则
- 每个部门有可执行的 Skill，而非停留在职责描述
- 七阶段流水线：问诊 → 规划 → 构建 → 审查 → 质检 → 发布 → 复盘
- 铁律：无调查不修 bug；无审查不发布
- 审查阶段采用多维度交叉：代码审查 + 安全审计 + 设计审查 + 根因分析 + 第二模型验证

### 全部 11 个 Skill
1. `/问诊` — 内阁需求官 — 6 问框架强制澄清痛点
2. `/审码` — 御史台代码官 — 6 维代码审查，严重问题自动修复
3. `/审安` — 兵部安全官 — OWASP + STRIDE 安全审计
4. `/审设` — 御史台设计官 — 设计一致性评分、AI Slop 检测
5. `/调查` — 御史台调查官 — 5 步法根因分析
6. `/codex` — 第二模型 — 独立交叉审查，发现盲区
7. `/质检` — 东厂质检官 — 浏览器自动化测试、6 类测试矩阵
8. `/性能` — 东厂性能官 — Core Web Vitals 基线、资源优化
9. `/发布` — 吏部发布官 — 版本管理、PR 创建、Changelog 生成
10. `/部署` — 吏部部署官 — 平台自动检测、部署执行、健康检查
11. `/复盘` — 东厂复盘官 — Git 统计、产出分析、趋势对比

## 任务状态机

```
待拆解 → 已拆解 → 审核中 → 已批准 → 执行中 → 待验收 → 已完成
                ↘ 封驳 → （回到已拆解，内阁重新规划）
         ↘ 已取消
```

合法状态转换：
- 待拆解 → 已拆解、已取消
- 已拆解 → 审核中
- 审核中 → 已批准、封驳
- 封驳 → 已拆解
- 已批准 → 执行中
- 执行中 → 待验收、已取消
- 待验收 → 已完成、执行中（打回重做）
- 已完成、已取消 → 终态

## 监控与复盘

- 东厂定期检查是否违反 Skill 优先、翻译规则、artefact 完成度等
- 御史台审核各部关键产出，维护质量底线
- 发现问题时在会话中提示，并在必要时更新相关说明文件

## 示例应用场景

- 自动化内容生产流水线（选题 → 资料收集 → 写稿 → 审核 → 发布）
- 简单财务统计与成本分析
- 技术文档批量生成与维护
- 项目管理与多 Agent 协作
