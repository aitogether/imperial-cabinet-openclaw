# AGENTS.md - 内阁都察制架构总览 v2

> **使用说明**: 复制为 `AGENTS.md` 放入你的 OpenClaw workspace，按需修改。

## 总体结构

| 角色 | 代号 | 说明 |
|------|------|------|
| 用户 | - | 唯一人类决策者 |
| 御前首辅（Grand Secretary） | main | 总调度，用户唯一接口 |
| 八部 | cabinet, engineering, finance, security, ops, content, compliance, docs | 八个职能部门 |
| 一厂 | monitor | 东厂（监控与审计） |
| 御史台 | censor | 都察院（专职审核与封驳） |

> **注**: 以上称呼仅为软件内部代号，不指代任何现实机构。

## 各部职责

### 内阁 (cabinet)
- **职责**: 总体协调与路线规划
- **具体**: 任务拆解、阶段目标与里程碑设计、验收标准制定、跨部门协调

### 工部 (engineering)
- **职责**: 技术选型与系统架构
- **具体**: 脚本和工具链实现、自动化流程与部署方案、性能优化

### 户部 (finance)
- **职责**: 算力与 API 成本统计
- **具体**: 时间投入与收益估算、预算 / ROI 分析与优先级建议

### 兵部 (security)
- **职责**: 安全策略与权限设计
- **具体**: 密钥和敏感信息保护、错误处理 / 备份与恢复 / 应急预案

### 吏部 (ops)
- **职责**: 流程与协作规范
- **具体**: 多 Agent workflow 与 SOP 设计、看板与任务流转规则、日常运营节奏

### 礼部 (content)
- **职责**: 对外文案与知识整理
- **具体**: 公告 / 教程、知识库结构、风格统一与表达优化

### 刑部 (compliance)
- **职责**: 隐私 / 版权 / 政策红线检查
- **具体**: 可做 / 不可做清单、合规版操作指引

### 翰林院 (docs)
- **职责**: README 与技术文档
- **具体**: 设计说明、教程与操作手册、复盘与经验沉淀

### 东厂 (monitor)
- **职责**: 监控与审计
- **具体**: 记录关键操作与决策链路、日志和指标采集、异常检测、定期复盘

### 御史台 (censor)
- **职责**: 审核与封驳
- **具体**: 审核各部关键产出质量，可封驳打回重做，维护整体质量与风险底线

## 标准工作流（七阶段流水线）

内阁都察制 v2 将所有任务拆为七个阶段，每个阶段由对应部门主导、具体 Skill 驱动：

```
问诊 → 规划 → 构建 → 审查 → 质检 → 发布 → 复盘
```

### 阶段 1：问诊 — 内阁主导

| Skill | 角色 | 功能 |
|-------|------|------|
| `/问诊` | 内阁需求官 | 6 问框架强制澄清痛点，输出设计文档 |

### 阶段 2：规划 — 工部主导

| 步骤 | 角色 | 功能 |
|-------|------|------|
| 内阁拆解 | 首辅 | 任务拆解、里程碑、验收标准 |
| 工部方案 | 工部 | 技术选型、架构设计、数据流、测试矩阵 |

### 阶段 3：构建 — 工部执行

| Skill | 角色 | 功能 |
|-------|------|------|
| coding-agent | 工部 | 代码实现、脚本编写（通过 Claude Code / Codex 等子代理） |

### 阶段 4：审查 — 御史台 + 兵部（四路并行）

> **统筹优化**：审码/审安/审设/codex 互不依赖，同时执行，提速 3 倍。

| Skill | 角色 | 功能 | 执行方式 |
|-------|------|------|---------|
| `/审` | 首辅编排 | 审查并行编排：一键触发四路审查 + 汇总裁决 | **编排入口** |
| `/审码` | 御史台代码官 | 6 维代码审查，严重问题自动修复 | 并行子代理 |
| `/审安` | 兵部安全官 | OWASP + STRIDE 安全审计，攻击场景验证 | 并行子代理 |
| `/审设` | 御史台设计官 | 设计一致性审查、AI Slop 检测 | 并行子代理 |
| `/codex` | 第二模型 | 独立交叉审查，发现盲区 | 并行子代理 |
| `/调查` | 御史台调查官 | 根因分析，5 步法追溯，铁律：无调查不修 | 按需触发 |

### 阶段 5：质检 — 东厂主导

| Skill | 角色 | 功能 |
|-------|------|------|
| `/质检` | 东厂质检官 | 浏览器自动化测试、6 类测试矩阵、回归测试 |
| `/性能` | 东厂性能官 | Core Web Vitals 基线、资源优化建议 |

### 阶段 6：发布 — 吏部 + 工部

| Skill | 角色 | 功能 |
|-------|------|------|
| `/发布` | 吏部发布官 | 版本管理、PR 创建、Changelog 生成 |
| `/部署` | 吏部部署官 | CI/CD 触发、线上验证、健康检查 |

### 阶段 7：复盘 — 东厂 + 翰林院

| Skill | 角色 | 功能 |
|-------|------|------|
| `/复盘` | 东厂复盘官 | 产出统计、测试覆盖率、问题清单、跨项目横向对比 |
| 文档更新 | 翰林院 | 同步文档、更新 README |

## 任务状态机

```
待拆解 → 已拆解 → 审核中 → 已批准 → 执行中 → 待验收 → 已完成
                ↘ 封驳 → （回到已拆解，内阁重新规划）
         ↘ 已取消
```

### 状态与阶段对应

```
待拆解 = 问诊前
已拆解 = 问诊完成，规划中
审核中 = 构建完成，审查阶段（/审 四路并行：审码+审安+审设+codex）
已批准 = 审查通过
执行中 = 质检 + 发布中
待验收 = 发布完成，待用户确认
已完成 = 复盘完成
```

### 合法状态转换白名单

| 当前状态 | 可转为 |
|---------|--------|
| 待拆解 | 已拆解、已取消 |
| 已拆解 | 审核中 |
| 审核中 | 已批准、封驳 |
| 封驳 | 已拆解 |
| 已批准 | 执行中 |
| 执行中 | 待验收、已取消 |
| 待验收 | 已完成、执行中（打回重做） |
| 已完成 | （终态） |
| 已取消 | （终态） |

### 快速通道（跳过审核）

以下场景可以跳过「审核中」状态，直接从「已拆解」进入「执行中」：

- 简单查询类任务（查日程、查消息）
- 用户明确说「不用审」的指令
- 紧急任务（用户标记加急）

## 会话与路由规则

1. **用户只和御前首辅（main）对话**
2. **御前首辅根据任务阶段，调用对应 Skill**：
   - 新需求 → `/问诊`
   - 规划完成 → 工部方案
   - 代码完成 → `/审码` → `/审安` → `/审设`
   - 审查通过 → `/质检` → `/性能`
   - 质检通过 → `/发布` → `/部署`
   - 发布完成 → `/复盘`
3. **子代理在职责范围内可自治**，但重要决策和风险通过御前首辅再报用户
4. **跨职能任务由内阁负责整体拆解与节奏**，御前首辅负责协同形成闭环 workflow
5. **任何阶段发现严重问题可打回**：/审码 封驳 → 工部重做；/审安 封驳 → 兵部评估风险
6. **铁律**：无调查不修 bug；无审查不发布
