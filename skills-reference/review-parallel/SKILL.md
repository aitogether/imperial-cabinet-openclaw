# /审 — 审查并行编排（统筹方法·关键路径优化）

## 设计理念

> 华罗庚《统筹方法》：互不依赖的工序应当并行执行。
> 审码、审安、审设、codex 四项审查互不依赖，可同时执行。

```
传统串行：审码(30min) → 审安(20min) → 审设(15min) → codex(25min) = 90min
并行执行：max(30, 20, 15, 25) = 30min         提速 3 倍
```

## 前置条件

- 构建阶段已完成，有实际代码产出
- 代码已提交（Git commit）

## 执行流程

### Step 1：准备审查材料

确认审查对象：
- 代码路径：项目根目录
- 变更范围：`git diff HEAD~N` 或指定 commit range
- 审查上下文：问诊/规划阶段的设计文档（如有）

### Step 2：四路并行 Spawn

同时 spawn 四个子代理，每个子代理读取对应 SKILL.md 并独立审查：

```
┌─ 审码 (censor-code-review/SKILL.md) ─── 6维代码审查
│
├─ 审安 (security-audit/SKILL.md) ─────── OWASP + STRIDE 安全审计
│
├─ 审设 (censor-design-review/SKILL.md) ── 设计一致性 + AI Slop 检测
│
└─ codex (codex/SKILL.md) ──────────────── 第二模型交叉审查
```

**Spawn 模板**：

```
# 子代理 1：审码
task: |
  你是御史台代码官。请阅读 skills/censor-code-review/SKILL.md，
  然后对以下代码执行 6 维审查：
  [代码路径/变更内容]
label: review-code

# 子代理 2：审安
task: |
  你是兵部安全官。请阅读 skills/security-audit/SKILL.md，
  然后对以下代码执行 OWASP + STRIDE 安全审计：
  [代码路径/变更内容]
label: review-security

# 子代理 3：审设
task: |
  你是御史台设计官。请阅读 skills/censor-design-review/SKILL.md，
  然后对以下代码/设计执行设计一致性审查：
  [代码路径/变更内容]
label: review-design

# 子代理 4：codex
task: |
  你是独立审查者。请阅读 skills/codex/SKILL.md，
  然后对以下代码独立执行交叉审查（不参考其他审查结论）：
  [代码路径/变更内容]
label: review-codex
```

### Step 3：等待与收集

- 四个子代理并行执行，互不知晓彼此
- 首辅通过 `sessions_yield` 等待全部完成
- 任一子代理超时（>5min）标记为"未完成"，不影响其他

### Step 4：汇总与决策

收到全部报告后，首辅汇总：

```markdown
# 审查汇总报告

## 四路审查结果

| 审查 | 结论 | 严重问题数 | 盲区(仅 codex 发现) |
|------|------|-----------|-------------------|
| 审码 | ✅通过 / ⚠️建议 / ❌打回 | N | — |
| 审安 | ✅通过 / ⚠️有条件 / ❌不通过 | N | — |
| 审设 | ✅通过 / ⚠️建议 / ❌打回 | N | — |
| codex | 一致性评估 + 盲区 | — | N 个盲区 |

## 交叉验证
- 四路一致的高置信度问题：N 个
- 仅 codex 发现的盲区：N 个（重点关注）
- 冲突结论：（如有，首辅裁决）

## 终审裁决
- 全部 ✅ → 进入质检阶段
- 有 ⚠️ → 有条件通过，建议修复清单附后
- 有 ❌ → 封驳打回工部重做

## 必须修复项（合并去重）
1. [严重] ...
2. [一般] ...

## 建议改进项
1. ...
```

### Step 5：执行决策

根据终审裁决：
- **通过**：更新任务状态为"已批准"，进入质检阶段
- **有条件通过**：附带修复清单，工部修复后无需重新审查
- **封驳**：更新任务状态为"封驳"，打回工部，说明原因

## 统筹优化点

### 并行收益

| 优化 | 传统 | 统筹后 | 收益 |
|------|------|--------|------|
| 审查阶段 | 串行 4 步 (90min) | 并行 1 步 (30min) | **3x 提速** |
| codex 等待 | 等首辅审完才开始 | 与首辅同时进行 | **零等待** |
| 汇总决策 | 逐个看完再判 | 一次性对比 | **减少上下文切换** |

### 关键路径识别

```
问诊 → 规划 → 构建 → 审查(并行) → 质检 → 发布 → 复盘
                     ↑ 瓶颈
```

审查是不可跳过的"烧水"步骤，但通过并行化，将 4 个"洗壶"步骤与"烧水"重叠。

### 进一步优化（"洗壶"前置）

| 可提前做的事 | 时机 | 效果 |
|-------------|------|------|
| 审设可先检查设计稿 | 构建阶段中 | 审查时只做代码级验证 |
| CI 编译验证 | commit 时自动跑 | 审码不浪费在编译错误 |
| 依赖安全扫描 | 问诊阶段 | 审安已有扫描结果 |

## 铁律

- **四路独立**：各审查子代理互不知晓，避免锚定效应
- **codex 盲区优先**：仅 codex 发现的问题是最有价值的
- **汇总归首辅**：最终裁决权在御前首辅，不由子代理定
- **封驳必说因**：打回时必须附具体问题清单
