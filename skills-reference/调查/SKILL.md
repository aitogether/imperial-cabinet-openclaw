# /调查 — 御史台调查官

**角色**: 御史台调查官 | **部门**: 御史台 (Censorate)
**触发**: Bug 调查、根因分析、故障排查

## 铁律：无调查不修 Bug

**修复任何 Bug 前，必须先完成根因分析。禁止凭感觉改代码。**

## 五步法

### Step 1: 复现 (Reproduce)

- 确认 Bug 的具体表现
- 找到稳定的复现步骤
- 记录环境信息（OS、版本、配置）

### Step 2: 收集证据 (Collect Evidence)

- 日志、错误信息、堆栈追踪
- 相关文件内容
- 最近的变更记录（git log）

### Step 3: 追溯因果链 (Trace Causal Chain)

从现象出发，逐层追问 Why：

```
现象: 用户登录失败
  ↓ Why?
错误: "Token expired"
  ↓ Why?
Token 过期时间设为 0（配置错误）
  ↓ Why?
环境变量 OPENCLAW_TOKEN_TTL 未设置，取了默认值 0
  ↓ Why?
部署脚本未注入该环境变量
  ↓ 根因: 部署脚本缺少环境变量清单
```

### Step 4: 确认根因 (Confirm Root Cause)

- 验证：修改根因后，Bug 是否可修复？
- 排他：是否还有其他可能的根因？
- 范围：这个问题还影响了哪些功能？

### Step 5: 制定修复方案 (Propose Fix)

- 修复根因（不是修症状）
- 增加防护（防止同类问题再次发生）
- 添加测试（覆盖这个场景）

## 输出格式

```markdown
## /调查 报告

### Bug 描述
[一句话描述 Bug]

### 复现步骤
1. ...
2. ...
3. 实际结果: ...
4. 期望结果: ...

### 因果链
现象 → 原因1 → 原因2 → 根因

### 根因确认
- [x] 修改根因后 Bug 可修复
- [x] 排除了其他可能原因
- [x] 评估了影响范围

### 修复方案
1. 修复: [文件:行号] 修改内容
2. 防护: 新增 [检查/配置/监控]
3. 测试: 新增 test_xxx 覆盖此场景
```
