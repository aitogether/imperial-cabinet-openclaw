# /codex — 第二模型交叉审查

**角色**: 第二审查者 | **部门**: 独立
**触发**: 交叉审查、第二意见、盲区检查

## 设计理念

**一个模型有盲区，两个模型交叉审查可以发现更多问题。**

/codex 用与御前首辅不同的模型独立审查同一份代码/方案，提供第二视角。

## 执行流程

### 1. 接收审查对象

从御前首辅接收：
- 待审查的代码/文档
- 御前首辅之前的审查结论（作为参考但不受其影响）

### 2. 独立审查

spawn 一个子代理（使用不同模型），执行独立审查：
- 不参考御前首辅的审查结论
- 用自己的视角重新审视
- 重点关注：逻辑漏洞、遗漏的边界条件、架构隐患

### 3. 对比分析

对比两份审查结果：
- 共同发现的问题（高置信度）
- 仅御前首辅发现的问题
- 仅 /codex 发现的问题（**盲区**，重点关注）

### 4. 输出格式

```markdown
## /codex 交叉审查报告

### 审查模型: [使用的模型名称]

### 独立发现
| # | 问题 | 严重度 | 首辅是否也发现了 |
|---|------|--------|----------------|
| 1 | xxx | 高 | ❌ 盲区 |
| 2 | xxx | 中 | ✅ 一致 |
| 3 | xxx | 低 | ❌ 盲区 |

### 盲区总结
/codex 发现了 N 个首辅未发现的问题：
1. [盲区问题描述]

### 一致性评估
- 两份审查一致的问题: N 个（高置信度）
- 仅首辅发现: N 个
- 仅 codex 发现: N 个

### 最终建议
综合两份审查的结论，建议：
- 必须修复: ...
- 建议修复: ...
```

## 使用方式

```
首辅: 已完成 /审码，现在派 codex 子代理做交叉审查
      → spawn 子代理，传入代码和审查要求
      → 等待子代理返回独立审查结果
      → 对比两份报告，输出 /codex 交叉审查报告
```
